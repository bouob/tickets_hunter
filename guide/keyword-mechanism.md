## 核心概念

### 什麼是「關鍵字匹配」？

**關鍵字匹配**是程式自動選擇日期或區域的主要方式。您只需提供**部分文字**，程式會自動找到包含該文字的選項。

**舉例**：
- 網站顯示：「2024/11/16 (六) 19:30」
- 您的關鍵字：`"11/16"` 或 `"週六"` 或 `"19:30"`
- 結果：程式會自動選擇這個日期

---

### 什麼是「回退策略」？

**回退策略**是當關鍵字找不到時，程式會自動改用的備用方案。

**三層回退機制**：
```
第 1 層：嘗試關鍵字匹配
   ↓ 找不到
第 2 層：使用 mode 自動選擇（from top/bottom/center/random）
   ↓ 找不到
第 3 層：等待手動操作或重試
```

這確保即使關鍵字設定錯誤，程式仍能繼續執行。

### 設定範例

#### 範例 1：指定特定日期
**在圖形介面中設定**：
- 日期自動點選：啟用
- 日期關鍵字：11/16
- 日期排序方式：from top to bottom

**執行邏輯**：
1. 先找包含 11/16 的日期
2. 找到 → 選擇該日期
3. 找不到 → 選第一個可用日期

---

#### 範例 2：多組關鍵字（OR 邏輯）
**在圖形介面中設定**：
- 日期關鍵字：11/16;11/17;週六

**執行邏輯**：
1. 先找包含 11/16 的日期 → 找到就選
2. 找不到，再找 11/17 → 找到就選
3. 還是找不到，再找 週六 → 找到就選
4. 全部找不到 → 使用日期排序方式自動選

---

#### 範例 3：不指定關鍵字
**在圖形介面中設定**：
- 日期關鍵字：留空

**執行邏輯**：
- 直接使用日期排序方式自動選擇
- 例如 from top to bottom 會選第一個日期

### 進階技巧

#### 技巧 1：指定時間段
**日期關鍵字設定**：19:30;晚上

優先選擇晚上場次。

#### 技巧 2：指定星期
**日期關鍵字設定**：週六;週日

優先選擇週末場次。

#### 技巧 3：空格 AND 邏輯（單一關鍵字內）
**日期關鍵字設定**：週六 19:30

必須同時包含「週六」**和**「19:30」才匹配。

#### 技巧 4：多組 OR 邏輯
**日期關鍵字設定**：11/16 上午;11/16 下午

先找「11/16 上午」，找不到再找「11/16 下午」（依序嘗試）。

**注意**：逗號僅作為千位分隔符號（例如票價 "3,280" 中的逗號），不用於分隔多組關鍵字。

## 自動回退策略

### 四種 Mode 說明

#### 1. from top to bottom（從上到下）
**在圖形介面中選擇**：日期排序方式 / 區域排序方式 → from top to bottom

**行為**：選擇**第一個**可用選項

**適用情境**：
- **日期**：選最早場次
- **區域**：選最貴/最好的位置（網站通常把貴的放前面）

**範例**：
```
可用選項：
  1. VIP區 $3880
  2. 搖滾A區 $2280
  3. 搖滾B區 $1980

→ 選擇：VIP區 $3880
```

---

#### 2. from bottom to top（從下到上）
**在圖形介面中選擇**：日期排序方式 / 區域排序方式 → from bottom to top

**行為**：選擇**最後一個**可用選項

**適用情境**：
- **日期**：選最晚場次
- **區域**：選最便宜/最後面的位置

**範例**：
```
可用選項：
  1. VIP區 $3880
  2. 搖滾A區 $2280
  3. 搖滾B區 $1980

→ 選擇：搖滾B區 $1980
```

---

#### 3. center（中間）
**在圖形介面中選擇**：日期排序方式 / 區域排序方式 → center

**行為**：選擇**中間**的選項

**適用情境**：
- 不想選太貴也不想選太便宜
- 平衡策略

**範例**：
```
可用選項：
  1. VIP區 $3880
  2. 搖滾A區 $2280
  3. 搖滾B區 $1980

→ 選擇：搖滾A區 $2280
```

---

#### 4. random（隨機）
**在圖形介面中選擇**：日期排序方式 / 區域排序方式 → random

**行為**：**隨機**選擇一個可用選項

**適用情境**：
- 增加不可預測性
- 分散風險
- 多人同時搶票時避免衝突

**範例**：
```
可用選項：
  1. VIP區 $3880
  2. 搖滾A區 $2280
  3. 搖滾B區 $1980

→ 選擇：（每次執行結果不同）
```

## 早期返回模式與條件式遞補 (Feature 003)

**版本**: 1.0 | **實作日期**: 2025-11-01
**實作平台**: TixCraft ✅, KKTIX ✅ | **待實作**: iBon, TicketPlus, KHAM, FamiTicket

---

### 🎯 什麼是早期返回模式？

**早期返回模式 (Early Return Pattern)** 是 Feature 003 引入的核心改進，改變了關鍵字匹配的執行方式。

#### 舊邏輯 (v1.0 之前)
```
1. 掃描所有關鍵字
2. 收集所有匹配結果
3. 從匹配清單中選擇一個
```

**問題**：
- ❌ 即使第一個關鍵字已匹配，仍會繼續檢查所有關鍵字
- ❌ 浪費約 30% 的處理時間
- ❌ 關鍵字順序沒有實質意義

#### 新邏輯 (v1.0+)
```
1. 依優先順序檢查第一個關鍵字
2. ✅ 匹配成功 → 立即選擇並停止
3. ❌ 匹配失敗 → 繼續下一個關鍵字
4. 重複步驟 1-3 直到找到匹配或全部失敗
```

**優點**：
- ✅ 第一個關鍵字匹配時立即停止（節省 ~30% 時間）
- ✅ 關鍵字順序決定優先權
- ✅ 更符合用戶的直覺預期

### 📊 實際影響對比

**設定**：日期關鍵字 = `11/16;11/17;11/18`

**可用日期選項**：

| **新邏輯** | 檢查 11/16 ✅<br>**立即停止** | 選擇 11/16 | **~33%** ⚡ |

---

#### 範例場景 B：多組區域關鍵字
**設定**：區域關鍵字 = `搖滾A;搖滾B;VIP`

**可用區域選項**：

| **新邏輯** | 檢查 搖滾A ✅<br>**立即停止** | 選擇 搖滾A區 $2280 | **~33%** ⚡ |

**重要**：新邏輯下，**關鍵字順序決定優先權**！

### 🔧 關鍵字順序的重要性

在早期返回模式下，關鍵字的排列順序變得**至關重要**。

#### ✅ 正確範例 1：日期由具體到模糊

**日期關鍵字設定**：`11/16 19:30;11/16;週六`

**邏輯**：
1. 優先嘗試最精確的匹配：「11/16 且 19:30」
2. 找不到，放寬條件：「11/16」
3. 還是找不到，再放寬：「週六」

**結果**：
- 若存在「11/16 19:30」→ 直接選擇（最佳匹配）
- 若只有「11/16 14:00」→ 選擇這個（次佳）
- 若只有「11/17 (六)」→ 選擇這個（備選）

---

#### ✅ 正確範例 2：區域由具體到模糊

**區域關鍵字設定**：`搖滾A 前排;搖滾A;搖滾B`

**邏輯**：
1. 優先嘗試最精確的匹配：「搖滾A 且 前排」
2. 找不到，放寬條件：「搖滾A」
3. 還是找不到，再放寬：「搖滾B」

**結果**：
- 若存在「搖滾A 前排」→ 直接選擇（最佳匹配）
- 若只有「搖滾A 後排」→ 選擇這個（次佳）
- 若只有「搖滾B」→ 選擇這個（備選）

---

#### ❌ 錯誤範例 1：日期順序顛倒

**日期關鍵字設定**：`週六;11/16;11/16 19:30`

**問題**：
- 第一個關鍵字「週六」太模糊
- 一旦匹配到任何週六的日期，就會**立即停止**
- 後面的「11/16 19:30」永遠不會被檢查

**結果**：
- 可能選到「11/17 (六) 14:00」而非您想要的「11/16 (六) 19:30」

---

#### ❌ 錯誤範例 2：區域順序顛倒

**區域關鍵字設定**：`VIP;搖滾A 前排;搖滾A`

**問題**：
- 第一個關鍵字「VIP」太廣泛
- 一旦匹配到任何 VIP 區，就會**立即停止**
- 後面的「搖滾A 前排」永遠不會被檢查

**結果**：
- 可能選到「VIP區 後排 $3880」而非您想要的「搖滾A 前排 $2280」
- 花更多錢卻不是您最想要的位置

---

**建議原則**：**最想要的放前面，備選方案放後面**

### 🛡️ 條件式遞補機制

Feature 003 同時引入了**條件式遞補 (Conditional Fallback)**，讓您更精確控制當所有關鍵字都失敗時的行為。

#### 新增設定欄位

| 欄位名稱 | 類型 | 預設值 | 說明 |
|---------|------|-------|------|
| `date_auto_fallback` | 布林值 | **false** | 日期關鍵字全失敗時是否自動遞補 |
| `area_auto_fallback` | 布林值 | **false** | 區域關鍵字全失敗時是否自動遞補 |

---

#### 模式對比

**日期選擇**：

| 模式 | date_auto_fallback | 行為描述 |
|-----|-------------------|---------|
| **嚴格模式** | **false** (預設) | 關鍵字全失敗時**停止執行**，等待手動介入<br>**適用**：只想買特定日期，其他都不要 |
| **自動遞補模式** | **true** | 關鍵字全失敗時**自動遞補**，使用「日期排序方式」選擇<br>**適用**：能買到票比買到特定日期更重要 |

**區域選擇**：

| 模式 | area_auto_fallback | 行為描述 |
|-----|-------------------|---------|
| **嚴格模式** | **false** (預設) | 關鍵字全失敗時**停止執行**，等待手動介入<br>**適用**：只想買特定區域，其他都不要 |
| **自動遞補模式** | **true** | 關鍵字全失敗時**自動遞補**，使用「區域排序方式」選擇<br>**適用**：能買到票比買到特定區域更重要 |

#### 實際範例：嚴格模式 vs 自動遞補

##### 範例 A：日期選擇

**場景**：只想買 11/16 的票

**在圖形介面中設定**：
- 日期關鍵字：`11/16`
- 日期排序方式：`from top to bottom`

**情境 1：嚴格模式** (date_auto_fallback = **false**)
```
可用日期：
  1. 2024/11/15 (五) 19:30
  2. 2024/11/17 (日) 14:00

執行結果：
[DATE KEYWORD] Checking keyword #1: 11/16
[DATE KEYWORD] All keywords failed to match
[DATE FALLBACK] date_auto_fallback=false, fallback is disabled
[DATE SELECT] Waiting for manual intervention

→ 停止執行，不會選擇其他日期 ✋
```

**情境 2：自動遞補模式** (date_auto_fallback = **true**)
```
可用日期：
  1. 2024/11/15 (五) 19:30
  2. 2024/11/17 (日) 14:00

執行結果：
[DATE KEYWORD] Checking keyword #1: 11/16
[DATE KEYWORD] All keywords failed to match

→ 自動選擇第一個可用日期 (11/15) ✅
```

**風險警告**：
- ⚠️ 自動遞補模式可能選到您不想要的選項
- ⚠️ 預設為嚴格模式 (false) 以防止誤購

### 🎯 使用建議

#### 建議策略 1：保守策略（預設）

**設定**：
- `date_auto_fallback = false`
- `area_auto_fallback = false`

**適用情境**：
- 只想買特定日期/區域的票
- 寧可買不到也不要買錯
- 高價值演唱會或活動

---

#### 建議策略 2：積極策略

**設定**：
- `date_auto_fallback = true`
- `area_auto_fallback = true`

**適用情境**：
- 任何日期/區域都可接受
- 能買到票最重要
- 低風險活動或便宜票券

---

#### 建議策略 3：混合策略

**設定**：
- `date_auto_fallback = false`（日期嚴格）
- `area_auto_fallback = true`（區域寬鬆）

**適用情境**：
- 日期固定（例如配合休假）
- 區域彈性（哪一區都可以）

**範例**：
```
日期關鍵字：11/16
date_auto_fallback = false  → 只買 11/16，其他日期都不要

區域關鍵字：搖滾A;搖滾B;VIP
area_auto_fallback = true   → 優先這些區域，沒有的話其他區域也可以
```

## ❓ 常見問題

### Q1: 關鍵字要完全一樣嗎？
**A:** 不用！只要**包含**您的關鍵字即可。

**範例**：
- 網站顯示：「2024/11/16 (六) 19:30 場」
- 關鍵字：`"11/16"` ✅ 可匹配
- 關鍵字：`"週六"` ✅ 可匹配
- 關鍵字：`"19:30"` ✅ 可匹配

---

### Q2: 多組關鍵字是 AND 還是 OR？
**A:** 有三種邏輯：

**OR 邏輯**（分號分隔）：
- 日期關鍵字設定：11/16;11/17
- 找到 11/16 **或** 11/17 都可以

**空格 AND**（單一關鍵字內）：
- 日期關鍵字設定：週六 19:30
- 必須同時包含「週六」**且**「19:30」

**分號分隔**（多組關鍵字組合）：
- 日期關鍵字設定：11/16 上午;11/16 下午
- 分號表示「或」：先嘗試第一組（11/16 上午），如果失敗則嘗試第二組（11/16 下午）
- 每組內部用空格表示「且」：11/16 **且** 上午

---

### Q3: 如果關鍵字打錯了會怎樣？
**A:** 程式會自動**回退到日期/區域排序方式**，不會失敗。

**範例**：
- 日期關鍵字：11/99（不存在的日期）
- 日期排序方式：from top to bottom
- 結果：找不到 11/99，會自動選第一個可用日期

---

### Q4: keyword_exclude 可以用在日期嗎？
**A:** 目前排除關鍵字只適用於**區域選擇**。

如果想排除某些日期，建議：
- 使用更精確的日期關鍵字
- 或手動選擇

---

### Q5: 可以看到匹配過程嗎？
**A:** 可以！在圖形介面的「進階設定」中啟用「輸出詳細除錯訊息」。

**警告**：僅當您需要除錯時使用，輸出大量日誌可能影響效能。

您會看到類似輸出：
```
[DATE KEYWORD] Trying keyword: "11/16"
[DATE KEYWORD] Match found! Total: 1
[DATE SELECT] Selected: 2024/11/16 (六) 19:30

[AREA KEYWORD] Trying keyword: "搖滾A"
[AREA KEYWORD] Match found! Total: 1
[AREA SELECT] Selected: 搖滾A區 $2280
```

---

### Q6: KKTIX 票券顯示「尚未開賣」會怎樣？
**A:** 系統會智能處理「尚未開賣」的票券，並持續監控直到開賣。

**行為說明**：
1. **關鍵字匹配**：即使票券顯示「尚未開賣」，系統仍會進行關鍵字匹配
2. **追蹤待開賣票券**：匹配成功的「尚未開賣」票券會被追蹤並顯示在日誌中
3. **自動刷新**：系統會自動刷新頁面，直到票券開始販售

**Verbose 模式輸出範例**：
```
[KKTIX AREA] Match Summary:
  Tickets matched (with input): 0
  Tickets matched (waiting for open): 1

  Waiting for these tickets to open:
    - 全票(2F) TWD$3,800 (keywords: 2F, 3,800)
```

**為什麼需要這個功能**：
- 避免您誤以為關鍵字設定錯誤
- 清楚顯示系統正在等待哪些票券開賣
- 一旦票券開賣，系統會立即執行購買

**支援語言**：繁體中文（尚未開賣）、英文（Not Started）、日文（まだ発売）
